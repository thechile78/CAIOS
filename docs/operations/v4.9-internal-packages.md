# CAIOS v4.9 — Immutable internal packages

## Purpose

Convert an approved internal handoff into a durable package snapshot without contacting WordPress or any other publishing platform.

## Safety guarantees

- Only administrators, editors, and producers may package a handoff.
- The handoff must be fresh and in `ready_for_packaging` or `packaging`.
- The story must still be approved.
- All editorial checks and human approval must remain complete.
- A matching immutable approval record must exist.
- One package is permitted per handoff.
- Package rows have read-only RLS; no authenticated update or delete policy exists.
- Story, checklist, approval, and source data are captured in the same database transaction that marks the handoff packaged and writes the audit event.

## Staging verification

1. Apply migrations through v4.9 to a non-production Supabase project.
2. Create test users for each application role.
3. Create and approve a test story through the normal CAIOS workflow.
4. Request an internal handoff.
5. Confirm unauthorized roles cannot package it.
6. Confirm stale `updated_at` tokens fail.
7. Confirm incomplete checklist or missing approval records fail.
8. Create the package and verify the snapshot and audit event.
9. Confirm a second package attempt fails.
10. Confirm ordinary authenticated users cannot update or delete the package row.

## Explicit exclusions

This milestone does not add WordPress credentials, WordPress API calls, public draft creation, scheduling, publishing, deployment, or browser access to a service-role key.
