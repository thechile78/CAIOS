# CAIOS v4.4 Authenticated Editorial Queue

This milestone begins replacing static newsroom data with authenticated Supabase reads. The command center loads active stories through the signed-in user's session, so the database row-level security policies remain the authorization boundary.

## Included

- Server-only story repository
- Authenticated read of active, non-archived stories
- Result limit constrained to 1–100 records
- Role capability helper for future story creation
- Queue empty state and role-aware messaging
- No service-role access

## Safety model

The command center still requires an authenticated, active profile before the queue is loaded. The repository uses the public anon key plus the user's session cookies; it does not bypass RLS. Database errors return a generic application error and do not expose query details.

Story creation is intentionally not enabled in this slice. The next slice must add validated server actions, database transactions or an RPC that records the story and audit event together, and staging tests for every role.

## Staging verification

1. Apply the v4.2 migrations to a non-production Supabase project.
2. Configure the v4.3 authentication environment variables.
3. Create test stories under multiple desks and statuses.
4. Verify anonymous and inactive users cannot access the command center.
5. Verify authenticated users see active stories but not archived stories.
6. Verify no service-role key is required.
7. Confirm no WordPress, scheduling, or publishing action is available.
