# CAIOS v4.6 Story Editing and Workflow Transitions

This milestone adds authenticated story editing without adding approval or publishing.

## Controls

- Only administrator, editor, producer, and researcher roles may open or submit the editor.
- The database derives the actor from `auth.uid()`.
- The client submits the version timestamp it loaded.
- The PostgreSQL function locks the row and rejects stale writes.
- Story changes and the matching audit event occur in one transaction.
- Input lengths and priority values are validated in both the application and database layers.
- Workflow transitions use an explicit allow-list.
- Approved, WordPress draft, published, and archived stories are locked from this editor.

## Staging verification

1. Apply all migrations to a non-production Supabase project.
2. Create two editor sessions and open the same story in both.
3. Save from the first session, then confirm the second receives the stale-version message.
4. Confirm invalid forward jumps are rejected.
5. Confirm valid forward and corrective backward transitions succeed.
6. Confirm every successful save produces exactly one `story_updated` audit event.
7. Confirm read-only and reviewer accounts cannot access or submit the editor.
8. Confirm approved and later-stage records cannot be edited through this workflow.

No WordPress connection, publication request, scheduling, deployment, service-role client, or production credentials are included.
