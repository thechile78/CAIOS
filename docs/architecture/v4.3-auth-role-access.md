# CAIOS v4.3 Authentication and Role-Aware Access

## Purpose

This milestone connects the CAIOS application shell to Supabase authentication without provisioning production infrastructure or enabling publishing. The newsroom now has a sign-in screen, server-side session validation, active-profile checks, role helpers, and fail-closed route protection.

## Session model

- Supabase authentication cookies are refreshed in `proxy.ts`.
- Server Components use `lib/supabase/server.ts`.
- Client Components may use `lib/supabase/browser.ts` only with the public URL and anon key.
- The service-role key remains server-only and is not used by normal user sessions.
- The command center calls `requireCurrentProfile()` and therefore requires both a valid Supabase user and an active `profiles` record.

## Roles

- `administrator`: platform and user administration
- `editor`: editorial management and approval
- `producer`: newsroom creation and editing
- `researcher`: research and source work
- `reviewer`: fact-check and approval decisions
- `read_only`: authenticated viewing only

Role checks are performed on the server. UI visibility is not an authorization boundary.

## Safe staging setup

1. Create a separate non-production Supabase project.
2. Apply the v4.2 migrations.
3. Store `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` in local or staging environment settings.
4. Do not add `SUPABASE_SERVICE_ROLE_KEY` unless a reviewed server-only operation requires it.
5. Configure the allowed redirect URL for `/auth/callback`.
6. Create test accounts for every role.
7. Confirm new accounts default to `read_only`.
8. Confirm disabled profiles and anonymous users cannot reach `/`.
9. Test sign-in, sign-out, expired sessions, and invalid callback codes.
10. Keep public registration disabled unless a separate security review approves it.

## Security boundaries

- Redirect destinations must be relative paths and cannot begin with `//`.
- Missing authentication configuration redirects protected routes to the login screen rather than exposing mock newsroom data.
- Missing, inactive, or malformed profiles receive no newsroom access.
- No WordPress credentials, deployment configuration, or publishing actions are included.
