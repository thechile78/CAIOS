# CAIOS v4.2 Database and Authentication Foundation

## Purpose

This milestone establishes the persistence and identity boundaries required for a secure newsroom application. It does not provision a live database, add production credentials, deploy CAIOS, or enable publishing.

## Selected foundation

CAIOS uses Supabase-compatible PostgreSQL because it provides PostgreSQL, managed authentication, migrations, and row-level security in one platform. The SQL remains standard PostgreSQL where practical so the data model is portable.

## Identity and roles

Every authenticated user receives a `profiles` row. New accounts default to `read_only`; an administrator must explicitly elevate access.

Roles:

- `administrator`: user and platform administration
- `editor`: editorial management and approval
- `producer`: newsroom creation and editing
- `researcher`: research, sources, and drafts
- `reviewer`: fact-check and approval decisions
- `read_only`: authenticated viewing only

## Core records

- `stories`: canonical newsroom items and workflow status
- `story_sources`: attribution and verification evidence
- `editorial_checklists`: mandatory editorial controls
- `approvals`: append-only human decisions
- `audit_events`: append-only activity history
- `publication_records`: controlled external publication requests and outcomes

## Security model

All newsroom tables have row-level security enabled. Anonymous access receives no policies and therefore no data. Authenticated access is constrained by role helper functions.

The `SUPABASE_SERVICE_ROLE_KEY` is a privileged server credential. It must exist only in server-side environment configuration. It must never use the `NEXT_PUBLIC_` prefix, appear in browser bundles, logs, screenshots, or committed files.

## Human Approval invariant

A story cannot hold `approved`, `wordpress_draft`, or `published` status without `approved_at` and `approved_by`. A publication record can only be requested when:

1. The requester is an administrator or editor.
2. The story is already approved.
3. Sources, facts, rights, SEO, and human approval checklist fields are complete.
4. A named approving user is recorded.

This database control supplements the application workflow; it does not replace server-side validation.

## Append-only records

Ordinary authenticated roles can insert approval and audit records but cannot update or delete them because no RLS policies grant those operations. Administrative correction should use a documented, audited database procedure rather than silently rewriting history.

## Safe rollout

1. Create a separate non-production Supabase project.
2. Record the project URL and anon key in local/staging secrets only.
3. Apply the migration using the Supabase migration workflow.
4. Create test users for every role.
5. Verify anonymous denial and least-privilege behavior.
6. Test backup and restore before importing real newsroom data.
7. Complete a security review before production provisioning.

## Rollback

This initial migration creates new objects and does not alter production data. Before rollback, export any test data that must be retained. Drop policies, triggers, functions, tables, and enum types in reverse dependency order. Never run destructive rollback SQL against production without a verified backup and explicit approval.

## Not included

- Live Supabase project provisioning
- Authentication UI or session middleware
- Supabase client packages
- Production data migration
- WordPress credentials or integration
- Automatic draft creation, scheduling, or publishing
- Deployment configuration
